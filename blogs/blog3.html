Building a Quiz Bank in Python with OOP

Posted on July 2, 2025

After rebuilding my coffee machine in Python using OOP, I wanted to take the concept further with another small but meaningful project. So I built a quiz bank system — a console-based app that delivers multiple-choice questions, scores the user, and provides feedback at the end.

But I didn’t just want to hardcode a bunch of questions into a loop. I wanted something modular, something I could easily extend later. So I structured the entire thing using object-oriented programming.

Why OOP for a Quiz App?

At first, a quiz may seem simple: just a list of questions and some logic to check answers. But when you look deeper, there are clearly separate components:

  • Questions (data)
  • Quiz logic (ask, check, track score)
  • Inputs and outputs (user interaction)

It made sense to break them into dedicated classes.

The Core Classes

I split the project into three key classes:

1. Question

This class stores the question text and the correct answer. Each instance represents one multiple-choice question.

2. QuizBrain

This is the core engine. It handles the logic for displaying questions, checking answers, and keeping track of the user's score and progress.

3. main.py

This is the driver script. It builds the question bank, initializes the QuizBrain instance, and starts the quiz loop.

How It Works

The app loops through each question in the bank, displays it, takes the user’s answer, and provides immediate feedback. At the end, it shows the final score. The questions can easily be loaded from a list or even an external API in the future.

What I Learned

  • How to use classes to represent real-world objects like questions and quizzes
  • Encapsulation — keeping data and logic tightly grouped
  • How OOP makes it easier to expand a program (e.g., adding question categories, levels, or different input sources)

Possible Improvements

  • Support for external JSON or CSV input for questions
  • GUI using Tkinter
  • Timer or difficulty settings
  • Track user history or progress

Final Thoughts

This project was small but solid. It helped me apply OOP principles to something more interactive and user-focused. It’s easy to build procedural scripts, but structuring your logic cleanly from the beginning sets the foundation for future expansion. That’s what I’m trying to practice.
